                                                           
● 현재 앱은 완전 결정론적 (알고리즘 열거 + 규칙 검증)이라, AI를 붙이기 가장 자연스러운 지점은 세 곳입니다.                      
  
  ---                                                                                                                             방향 1 — 자연어 파라미터 입력 (가장 실용적)                                                                                   
                                                                                                                                  사용자가 "13S5P, 엇배열, 좌측 B+ 출력, 금형 3종 이하" 같은 문장을 입력하면 Claude API가 {S, P, arrangement, b_plus_side, 
  max_plates} JSON으로 파싱해 rerender()에 전달.                                                                                

  트레이드오프: 구현이 가장 단순하고 기존 엔진을 그대로 유지. 단, LLM이 파라미터를 잘못 추론할 때 조용히 오작동할 수 있어 파싱
  결과를 UI에 표시하는 검증 단계가 필요.

  ---
  방향 2 — 검증 결과 설명 레이어

  validator.js의 pass/fail 결과를 Claude가 받아 "ICC①: G3의 rowSpan=3으로 원칙 16 위반 — 그룹을 2행 이내로 축소하세요" 같이
  한국어로 설명.

  트레이드오프: 엔지니어보다 비전공 의사결정자에게 유용. 설명 품질은 validator.js가 구조화된 오류 객체를 반환해야 좋아짐 (현재
  구조 확인 필요).

  ---
  방향 3 — AI 설계 조언자 (가장 강력하지만 범위 큼)

  후보 카드 목록을 AI에게 넘겨 "제조 난이도 / 비용 / BMS 배선 길이" 기준으로 순위와 이유를 출력. 현재의 점수
  공식(compactShapeScore)을 보완.

  트레이드오프: 매 탐색마다 API 호출이 발생해 비용·레이턴시가 누적됨. 탐색 완료 후 1회 호출로 제한하면 실용적.

  ---
  추천: 방향 1 → 방향 2 순서로 구현하는 게 투자 대비 효과가 큽니다. Claude API + tool_use로 파라미터 파싱을 강타입으로 처리하면
  오파싱 문제도 해결됩니다. 어느 방향으로 진행할지 정해주시면 바로 시작하겠습니다.

❯ 어떤 판단에 있어서, 인공지능이 들어간다면 더 좋지 않을까요?                                                                   

● 현재 시스템에서 판단이 필요한 지점 을 짚어보면:
                                                                                                                                
  ---                                                                                                                             현재 알고리즘이 하는 것 (규칙 기반, AI 불필요)
                                                                                                                                
  - ICC 통과/실패 — 이진 규칙
  - compactShapeScore 점수 계산 — 고정 공식
  - 후보 열거 — 완전 탐색

  ---
  AI 판단이 의미 있는 지점

  ① 후보 0개일 때 원인 진단
  현재: 그냥 "배열을 만들 수 없습니다" 표시.
  AI: "holder_rows=3이 너무 작습니다. 4로 늘리거나 max_plates 제한을 풀면 후보가 생깁니다." — 파라미터 조정 방향을 구체적으로   
  제안.

  ② 후보 선택 조언
  현재: 점수 순서로 카드 나열, 사용자가 직접 고름.
  AI: "금형 비용 우선이면 카드 2번, 조립 난이도 우선이면 카드 4번 — 이유는 …" — 맥락 기반 추천.

  ③ ICC 경계값 근처 설계 경고
  현재: rowSpan=2이면 통과, rowSpan=3이면 탈락 — 경계 없음.
  AI: "rowSpan=2인 그룹이 4개 — 설계는 통과했지만 공차 여유가 없어 제조 리스크가 있습니다."

  ---
  가장 ROI가 높은 한 곳을 고르라면 → ①번 (후보 0개 진단)입니다. 사용자가 가장 막히는 순간이고, 현재 시스템이 아무 정보도 주지   
  않는 지점이라 AI가 가장 티 납니다.
