**4. 3대 치명 리스크 및 완화 전략 \[v2.0 신규\]**

6인 전문가가 공통으로 식별한 가장 치명적인 리스크 3가지와 그 완화
전략입니다.

**리스크 1: 리뷰 단계 추가가 병목 증폭으로 전환**

문제: 서브에이전트 이중 리뷰가 \"구현 대기 → 리뷰어 대기 → 오너 대기\"의
3중 병목 생성 가능

**완화 전략: 3단계 점진 도입 + 자동화 최소 기준**

  ----------- ------------------- ------------------- -------------------
    **도입         **내용**            **조건**          **측정 기준**
    단계**                                            

   **1단계**  Inline Self-Review    즉시 적용 가능    tsc/lint/test PASS
               체크리스트만 적용                             확인

   **2단계**   spec-reviewer 1차   1단계 2주 운영 후  오너 리뷰 시간 측정
                    파일럿                            

   **3단계**  spec + code-quality 2단계 효과 검증 후  병목 발생 여부 판단
                   이중 리뷰                          
  ----------- ------------------- ------------------- -------------------

**리스크 2: 증거 기반 완료가 형식주의로 전락**

문제: tsc/lint/test 통과가 비즈니스 로직 정확성을 보장하지 않음

**완화 전략: Business AC + Technical AC 검증 항목 필수 추가**

- Technical AC (기술 검증, Necessary): tsc \--noEmit, npm test, lint
  PASS

- Business AC (비즈니스 검증, Sufficient): S/P 구성 일치, 셋 수 정합,
  원가 논리 검증, DFM PASS 확인

- 배터리팩 도메인 전용 AC: 조합도 이미지와 결과 교차검증

**리스크 3: 문서 해석 분열로 인한 SSOT 부재**

문제: 4개 문서에 규칙 분산, 충돌 발생 시 판단 불가

**완화 전략: 문서 우선순위 체계 + Layer별 변경 권한 명시**

**6. KMRA 프레임워크: 유지/수정/제거/추가 \[v2.0 신규\]**

전문가 합동 토론 4단계에서 도출된 통합 설계 재구성 결과입니다.

**6.1 Keep (절대 유지) --- 8개**

- 증거 기반 완료 원칙: AI 환각/완료 과장 차단의 핵심 (검증의 철의 법칙)

- 압박 저항 규칙 PR-1\~PR-5: 에이전트가 시간 압박에 품질 타협 방지

- Owner 승인 경계 명시화: Layer 1 문서는 반드시 Owner ECO 승인

- 문서 레이어 분리 철학: 불변/변동/임시 3계층

- 즉시/단기/중장기 로드맵 구조: 점진적 도입 원칙

- TDD (Test-Driven Development) Iron Law: 테스트 먼저 작성 후 구현
  (RED-GREEN-REFACTOR)

- Subagent 기반 개발: 독립 Git Worktree에서 병렬 작업 수행

- 코드 리뷰 프로세스: Superpowers 9-step 중 핵심 검증 단계 유지

**6.2 Modify (수정 필요) --- 5개**

- 이중 리뷰: 즉시 전면 도입 → 3단계 점진 도입 (1단계: Inline
  Self-Review부터)

- 검증 기준: 모든 태스크 동일 → Level A/B/C 차등화

- 정량 효과 수치: 확정적 표현 → KPI 가설 + 측정 필요로 재표기

- 문서 레이어별 변경 규칙: 개념만 → SSOT 우선순위 명시 추가

- 서브에이전트 입출력 형식: 미정의 → 표준 포맷 구체화 (## Verification
  Report)

**6.3 Remove (제거) --- 4개**

- 과도하게 확정적인 효과 표현 (\"즉시 차단\", \"40\~60% 단축\")

- 중복 서술 (완료 과장 금지, 증거 제출 반복)

- 모든 태스크 동일 강도 적용 암시

- 수작업으로도 충분하다는 낙관적 서술

**6.4 Add (신규 추가) --- 6개**

- Business AC + Technical AC 검증 항목 분리: Technical AC(기술 검증,
  Necessary) + Business AC(비즈니스 검증, Sufficient) 이원화

- 운영 KPI 가설 검증 체계: 완료 보고 반려율, 오너 리뷰 소요시간, 회귀
  버그 수

- 에스컬레이션 후 회복 플로우: 실패 → 분석 → 재시도 → 보고 프로세스

- 검증 결과 보고 표준 포맷: \## Verification Report + Evidence 구조

- 역할별 문서 분리 체계: CEO용/운영정책용/에이전트용/리뷰어용

- GC Agent (Garbage Collection Agent): 엔트로피 관리 백그라운드 에이전트
  --- 미사용 코드, 오래된 문서, 불필요한 의존성 자동 정리
