# Pack-Renderer 체계적 디버깅 규칙

> 목적: 증상 패치 반복을 멈추고 근본 원인을 제거한다.
> 호출 시점: 사용자가 버그·오류·"이상해"·"안 돼"·"왜 이래"를 보고할 때 / 본인이 기대와 다른 출력을 발견했을 때
> 최종 업데이트: 2026-04-17 (v1)

## 철칙 (Iron Laws)

1. **재현 불가능한 버그는 고치지 않는다.** 먼저 재현 방법을 확정한다.
2. **가설 없이 코드를 수정하지 않는다.** 먼저 "왜 그럴까"를 글로 쓴다.
3. **증상이 사라졌다고 해결된 것이 아니다.** 근본 원인을 명시하지 못하면 미해결이다.
4. **같은 파일을 3회 이상 같은 의도로 수정하면 즉시 멈춘다.** 접근이 틀렸다는 신호다.

---

## 5단계 디버깅 프로토콜

**1단계 — 재현 (Reproduce)**
- 버그의 **입력**, **실행 단계**, **기대 출력**, **실제 출력**을 명시한다.
- 재현이 불가능하면 사용자에게 정보를 요청한다. 추측으로 넘어가지 말 것.
- 재현 성공 = 실패하는 자동 테스트 1개 또는 재현 스크린샷 1장.

**2단계 — 격리 (Isolate)**
- 의심 범위를 좁힌다. 가능한 기법:
  - `console.log` + 이분 탐색 (bisection)
  - git bisect (언제부터 깨졌는지)
  - 입력을 최소 케이스로 축소 (S=13P=3 실패 → S=2P=1도 실패하는가?)
- 의심 함수를 **1개로 특정**할 때까지 2단계에 머문다.

**3단계 — 가설 (Hypothesize)**
- **코드 고치기 전에** "왜 이런 결과가 나오는지"를 1~3 문장으로 적는다.
- 2개 이상의 가설이 있으면 각각 기록하고 어느 것이 맞는지 먼저 확인한다.
- 가설이 "그냥 이쪽이 이상해서"처럼 모호하면 2단계로 복귀한다.

**4단계 — 검증 (Verify)**
- 가설을 확인할 **최소 실험**을 한다. 보통 1~3줄의 로그 또는 어서션 추가.
- 가설이 맞으면 5단계. 틀리면 3단계로 돌아가 새 가설을 세운다.
- **검증 없이 패치하는 것을 금지한다.**

**5단계 — 수정 + 회귀 방어 (Fix + Guard)**
- 근본 원인을 수정한다. **증상 우회(if 분기 추가, try-catch로 가리기) 금지.**
- 재현 테스트를 `tests/`에 추가한다 (tdd-first.md 규칙 준수).
- 전체 테스트 실행. 다른 항목이 깨졌으면 그 역시 근본 원인을 찾는다.
- 커밋 메시지에 **원인**을 명시한다. 증상만 적지 말 것.

---

## 증상 vs 근본 원인 — 판별 질문

수정안을 커밋하기 전에 스스로 묻는다:

1. 이 버그가 **왜** 발생했는지 한 문장으로 설명할 수 있는가?
2. 같은 원인으로 **다른 기능**이 깨질 가능성이 있는가? 있다면 그것도 수정했는가?
3. 내 패치가 원인을 제거한 것인가, **특정 입력에서만** 증상을 숨긴 것인가?
4. 1년 뒤의 내가 이 커밋을 보고 **왜 이 수정이 필요했는지** 알 수 있는가?

하나라도 "아니오"면 수정안을 버리고 2~3단계로 복귀한다.

## 금지된 안티패턴

- **Shotgun debugging**: 여러 군데를 동시에 고쳐서 "그중 뭔가 맞았겠지" 기대하기.
- **캐시 의심 남발**: 버그를 설명 못할 때 "브라우저 캐시일 거야"로 도망치기. 반드시 시크릿 모드·`?v=timestamp`·`Cache-Control` 로 배제 확인.
- **환경 탓 먼저 하기**: "Playwright 뷰포트 때문일 거야", "Windows DPI 때문일 거야" — 먼저 코드 가설을 배제하고 나서 환경을 의심한다.
- **CSS 6회 반복 수정 같은 땜질**: 세션 10 교훈. 3회 이상 같은 파일을 같은 의도로 수정하면 즉시 멈추고 가설을 재검토한다.
- **try-catch로 삼키기**: 에러를 보이지 않게 하는 것은 해결이 아니다.
- **"다른 곳에서는 잘 됨" 면피**: 내 환경에서 재현 안 된다고 버그가 없는 건 아니다.

## 사용자 버그 보고 수신 시 표준 질문

재현 정보가 부족하면 아래 3개를 먼저 묻는다:

1. **입력이 무엇입니까?** (S, P, arrangement, holder_rows/cols, 기타 UI 상태)
2. **어느 단계에서 증상이 나타납니까?** (렌더 직후 / 특정 버튼 클릭 후 / 특정 값 변경 후)
3. **실제로 본 것**과 **기대한 것**의 차이를 문장 또는 스크린샷으로.

추가로 필요 시:
- 콘솔 오류 메시지 전문
- `state/feature_list.json` 관련 항목 ID
- 어떤 커밋까지 정상이었는지

## 과거 사례 색인 (같은 실수 반복 금지)

- **세션 10 UI 이슈**: 코드(CSS)를 6회 수정했으나 원인은 Playwright 뷰포트 분리. → 3회 규칙 위반 사례. 다음부터 3회째에서 멈추고 환경을 의심한다.
- **face filter 버그 (세션 7, 커밋 98edee2)**: render() 본체에 faceFilter 미적용이 원인. 증상은 "Both/Top/Bottom 전환 시 화면 일부만 보임". → 체계적 격리로 원인 1회에 특정한 성공 사례.
